# Задача: заменить TOFU-якорь личности фото на consensus-anchoring (онбординг)

Начни в **Plan Mode**: это меняет продуктовый инвариант (фото-identity-гейт, PRODUCT_SPEC §1.3)
и, вероятно, требует изменения схемы Prisma. Сначала короткий план → дождись подтверждения,
потом правки. Перед изменением user-flow/инварианта спроси (правило AGENTS.md).

## Контекст
Gennety Dating — Telegram-first dating-бот (см. CLAUDE.md/PRODUCT_SPEC.md/ARCHITECTURE.md).
В онбординге есть стадия загрузки фото со strict-media валидацией (недавний коммит 58f9ecb):
safety + наличие лица + дедуп (perceptual hash) + identity-якорь. MIN_PHOTOS=2, MAX_PHOTOS=6,
PHOTO_BONUS_TICKET_THRESHOLD=4, FACE_SIMILARITY_THRESHOLD=0.60 (packages/shared/src/constants.ts).
Реальная привязка к владельцу — позже на Persona-верификации (Rekognition CompareFaces
селфи↔фото). Онбординг-якорь нужен только для внутренней консистентности (все фото = один человек).

## Баг (корень — подтверждён)
Пока пользователь не верифицирован (нет verifiedSelfiePath), identity-reference для нового фото
выбирается как `photos[0]` — ПЕРВОЕ фото (apps/bot/src/services/profile-media-validation/
profile-photo-validation.ts:66-73). Самое первое фото принимается вообще без identity-проверки
и через photoUploadStatePatch (apps/bot/src/services/profile-media-validation/photo-state.ts:43)
становится якорём (referenceFaceEmbedding из photos[0]). Это "trust-on-first-use":
- кто прислал фото первым — тот и объявлен владельцем;
- если прислать ЧУЖОЕ фото первым, якорем станет чужак, а РЕАЛЬНОЕ фото владельца будет
  отклонено как "не тот человек";
- из двух несовпадающих фото система НЕ может определить, какое из них владельца.

## Желаемое поведение: consensus-anchoring
Не доверять ни одному фото как якорю, пока нет ПОДТВЕРЖДЕНИЯ — минимум 2 загруженных фото
с ОДНИМ И ТЕМ ЖЕ лицом (совпадающий кластер). Самый большой взаимно-совпадающий кластер = владелец;
одиночки-выбросы отклоняются.

Правила:
1. ЕСЛИ пользователь уже верифицирован (есть verifiedSelfiePath) → НЕ менять: сравнение
   идёт с Persona-селфи (истинная личность).
2. ЕСЛИ не подтверждён (онбординг, селфи нет):
   - каждое входящее фото сначала проходит per-photo гейты (safe, ≥1 лицо, не дубликат) —
     они не зависят от личности;
   - прошедшие фото идут в ПУЛ КАНДИДАТОВ (pending): пока НЕ считаются в MIN_PHOTOS,
     НЕ попадают в Profile.photos[], НЕ рендерятся в профиле/карточках;
   - на каждое добавление — попарное сравнение кандидатов (Rekognition CompareFaces,
     ≥ FACE_SIMILARITY_THRESHOLD 0.60), строим кластеры (связные компоненты по рёбрам
     "лица совпали"; фото входит в кластер, если СОВПАДАЕТ ЛЮБОЕ его лицо — групповые фото
     остаются разрешены);
   - как только кластер достигает размера ≥2 → ПОДТВЕРДИТЬ личность:
       * выставить якорь (referenceFaceEmbedding) из кластера (репрезентант = самое раннее фото),
       * перенести фото кластера в принятые Profile.photos[] (+ photoFaceScores + hashes),
       * ОТКЛОНИТЬ все pending-фото вне кластера (другой человек) с понятным объяснением,
       * дальше — как сейчас (каждое новое фото сверяется с подтверждённым якорем);
   - если после нескольких pending-фото кластера ≥2 нет (все разные) → продолжать просить ещё,
     с явным сообщением; разумный cap на попытки, никогда не якорить одиночку.

3. Взаимодействие с MIN_PHOTOS: фото засчитывается в минимум только когда оно в подтверждённом
   кластере. Поэтому:
   - self, self → кластер {self,self} на 2 → подтверждено, минимум закрыт (3-е НЕ требуется);
   - self, stranger → кластера нет → "ничего не зафиксировано, пришли ещё" → self → кластер
     {self,self} подтверждён, stranger отклонён → минимум закрыт на 2 принятых;
   - то есть только при несовпадении первых двух пользователю нужно 3 загрузки ради 2 принятых.

## Краевые случаи (обязательно покрыть)
- 2 совпадающих фото → подтверждать СРАЗУ (не заставлять слать 3-е).
- 2 несовпадающих → не якорить, просить ещё, сказать что ничего пока не зафиксировано.
- 3+ все разные → кластера нет → продолжать просить + soft-cap + понятное сообщение.
- НИЧЬЯ: два кластера одинакового размера (self×2 + друг×2) — детерминированный tiebreak:
  брать кластер, сформированный РАНЬШЕ (самые ранние фото). Отметить как явное решение в плане;
  истинного владельца всё равно потом подтвердит Persona-верификация.
- Групповые фото: у фото может быть несколько лиц; "совпало" = совпало любое лицо. Сохранить
  правило "групповые фото разрешены, если владелец на них есть".
- Дедуп идёт ДО кластеризации: нельзя подтвердить кластер двумя копиями одного изображения
  (одинаковое фото дважды → дубликат отклоняется), значит кластер ≥2 = 2 РАЗНЫХ фото одного лица.
- Стоимость: попарный CompareFaces — O(n²), но только в неподтверждённой фазе и при MAX_PHOTOS=6
  это ≤~15 вызовов. После подтверждения попарных сравнений нет.

## Что менять
- Новый модуль, напр. services/profile-media-validation/identity-consensus.ts — кластеризация +
  машина состояний "pending → confirmed".
- profile-photo-validation.ts (выбор reference, строки 66-73) и photo-state.ts (подтверждение
  якоря) — сделать consensus-aware.
- ХРАНИЛИЩЕ pending-пула между сообщениями: НЕ загрязнять Profile.photos[]/photoFaceScores[]
  (сохранить инвариант photos[i] ↔ photoFaceScores[i]) и не показывать pending в профиле/карточках.
  Скорее всего нужна аддитивная колонка в schema.prisma (напр. Profile.pendingPhotos Json[])
  → это deploy-гейт: dev:db:push локально и db:push на проде (см. deploy.md; ср. с памятью про P2022).

  * apps/bot/src/handlers/onboarding/conversational.ts:1410 (Telegram онбординг)

## Ограничения / guardrails
- Не ломать: MIN_PHOTOS=2, MAX_PHOTOS=6, групповые фото, дедуп, safety/face-гейты,
  инвариант photos[i]↔photoFaceScores[i], Persona/Rekognition verification-пайплайн (без изменений —
  он и есть истинная привязка к владельцу), строгий TS (без any), без новых внешних зависимостей
  (переиспользовать существующий Rekognition CompareFaces).
- Backward-compat: профили с уже выставленным referenceFaceEmbedding не трогаем; новая логика
  работает только пока якоря нет (неподтверждён) и пользователь не верифицирован.

## Тесты и воспроизведение

несовпадении первых двух фото личность не фиксируется до прихода совпадающей пары; все 4
поверхности используют общую логику; тесты (вкл. "stranger-first") зелёные; PRODUCT_SPEC §1.3
и ARCHITECTURE обновлены; схема (если менялась) задокументирована как deploy-гейт.
