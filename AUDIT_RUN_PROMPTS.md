> (что читать, что проверять, какие инварианты). Здесь — **протокол выполнения**

>

> **найти → спланировать фиксы → исправить → перепроверить до конца.**

## Протокол выполнения (агент читает и следует)

Ты работаешь как **оркестратор полного цикла** для одного блока. Веди работу
через `TodoWrite` (по одной задаче на фазу и на каждый фикс) и не заканчивай ход,
пока не выполнены критерии готовности.

### Фаза 0 — Загрузка контекста
- Прочитай раздел нужного блока в `AUDIT_PLAN.md`: область, ключевые файлы,
  замысел, инварианты, чек-лист.
- Прочитай сами файлы блока **и их тесты**, установи design-intent из
  PRODUCT_SPEC/ARCHITECTURE. Код + тесты + `schema.prisma` — источник правды;
  где проза расходится с кодом, это находка.

### Фаза 1 — Планирование (аудит → план ремедиации)
- Проведи полный аудит строго по чек-листу блока: findings по severity
  (Critical/High/Medium/Low), каждая с `файл:строка`, конкретным сценарием
  провала (вход → неверный выход/краш) и пометкой, покрыта ли тестом.
- Для каждой находки составь **план фикса**: корневая причина, предлагаемое
  изменение, радиус поражения (что ещё трогает тот же сервис/callback/cron/поле),
  какой тест добавить/поправить, риск.
- Сгруппируй фиксы, упорядочь по зависимостям, зафиксируй план в `TodoWrite`.
- **Гейт согласования.** Если среди фиксов есть изменения продуктовых правил/
  user-flow, Prisma-схемы или векторных индексов, платёжной логики, или shape
  `/v1/*` (контракт iOS) — СНАЧАЛА покажи план и дождись подтверждения по этим
  пунктам (это прямое требование AGENTS.md «Ask first before»). Явные баги, гонки,
  утечки trust boundary, мёртвые ветки и недостающие тесты чини без ожидания.

### Фаза 2 — Оркестрация фиксов
- Реализуй фиксы наименьшим изменением, вписываясь в существующие границы и
  нейминг. При изменении поведения — **сначала тест** (или правка теста), потом код.
- Один логический фикс — один коммит (журнальное правило репозитория: `git add -A`
  → scoped commit → `git push origin HEAD` после каждого изменения; это durable,
  pre-authorized, спрашивать не нужно). Никогда не создавай ветки, никогда не
  стейджи `.env`/секреты/артефакты.
- Держи `TodoWrite` в актуальном состоянии: помечай фиксы done по мере закрытия.
- Не расширяй скоуп за пределы блока и его findings; не делай попутный рефакторинг
  несвязанного кода; уважай грязное рабочее дерево (не откатывай чужие правки).

### Фаза 3 — Повторная проверка (до готовности)
- Прогони **тот же чек-лист блока заново против нового кода** — убедись, что каждая
  находка реально закрыта и не появилось регрессий в смежных ветках.
- Запусти узкие тесты/typecheck для затронутых пакетов (команды из AGENTS.md, напр.
  `pnpm --filter @gennety/bot exec vitest run <файл>`, `pnpm --filter @gennety/bot
  typecheck`); расширяй охват, если риск оправдывает. Где есть runtime-поверхность —
  прогони поведение, а не только тесты.
- Зациклись: если перепроверка вскрыла новую проблему или неполный фикс — вернись
  в Фазу 2. Не останавливайся на «вроде поправил».
- Выполни Documentation Impact Check (AGENTS.md); обнови доки/Obsidian, только если
  затронуты инварианты/контракты/схема/деплой. Иначе явно скажи `Docs unaffected`.

### Критерии готовности (Definition of Done)
1. Каждая находка либо **исправлена, покрыта тестом и перепроверена**, либо
   **явно отложена** с одной строкой причины (нужно продуктовое решение / ждёт
   согласования схемы / вне скоупа блока).
2. Узкие тесты и typecheck затронутых пакетов зелёные (или падение явно названо
   в отчёте с причиной — журнал держим актуальным).
3. Каждый фикс закоммичен и запушен; Documentation Impact Check выполнен.
4. Итоговый отчёт: что найдено, что исправлено, что отложено и почему, остаточный
   риск, что осталось без тестового покрытия.

Если тесты/typecheck/build падают по причине, которую не удаётся закрыть в рамках
блока, всё равно коммить состояние (журнал), но громко назови падение в отчёте.

## Порядок

Рекомендованная последовательность — в конце `AUDIT_PLAN.md` («Порядок прохождения»):
сначала деньги/доверие (4 → 10 → 13 → 16 → 15 → 27), затем основной путь, затем
периферия. Один блок — одна сессия. Если фиксов много и они рискованные, дай агенту
пройти Фазу 1, утверди план, и только потом разреши Фазы 2–3.
